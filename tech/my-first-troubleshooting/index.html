Git Submodule 에러와 구글 애널리틱스 연동 해결기

May 26, 2026

1. 멘붕의 시작

블로그에 구글 애널리틱스를 달아서 방문자를 확인해 보려다가 에러 파티를 만났습니다. 터미널 창에 Repository not found 같은 무서운 빨간 글씨들이 쏟아지고 배포가 멈췄을 땐 정말 당황했죠. 하지만 천천히 원인을 찾고 해결해 나간 과정을 기록해 봅니다.

2. 도대체 왜 그랬을까?

찬찬히 살펴보니 제가 놓친 사소한 실수 세 가지가 겹쳐 있었습니다.

  • 폴더 경로 문제: 윈도우 바탕화면(한글 경로와 클라우드 연동)에서 작업하다 보니 프로그램들이 파일 위치를 잘 못 찾고 엉켰습니다.
  • 주소 오타: 깃허브 저장소를 만들 때 내아이디.github.io라고 정확히 써야 하는데, 실수로 github.io라고만 적는 바람에 터미널이 길을 잃었습니다.
  • 꼬여버린 순서: 깃허브 웹사이트에서 README.md 파일을 먼저 만들어버린 탓에, 내 컴퓨터와 인터넷 서버의 파일 순서가 어긋나서 밀어올리기(push)가 완전히 막혀버렸습니다.

3. 이렇게 해결했습니다!

작업 폴더를 윈도우의 가장 깔끔한 영문 경로(C:\workspace)로 옮기고, 꼬인 파일들을 하나씩 정리했습니다.

첫째, 꼬인 폴더 지우기 내 컴퓨터에 잘못 남아있던 찌꺼기 파일들을 명령어로 깨끗하게 지웠습니다.

rm -r -force public rm -r -force .git/modules/public

둘째, 올바른 주소로 다시 연결하기 오타 났던 주소를 고치고, 내 블로그와 깃허브 전시관을 엮어주는 명령어를 다시 입력했습니다.

git submodule add -b main https://github.com/xoxo787/xoxo787.github.io.git public

셋째, 강제로 배포하기 모든 세팅을 마치고 터미널에서 강제 배포 명령을 내려 꼬인 역사를 덮어썼습니다.

hugo cd public git push origin main

4. 느낀 점

처음 에러가 났을 땐 눈앞이 캄캄했지만, 에러 메시지를 천천히 읽어보고 오타 하나, 폴더 경로 하나를 꼼꼼히 맞추는 게 얼마나 중요한지 깨달았습니다. 드디어 애널리틱스가 연동되고 배포에 성공하니 너무 뿌듯하네요!
